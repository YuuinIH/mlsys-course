# 贡献指南

这个仓库还在 Alpha 草稿阶段，最需要的是清楚、可复现、能解释现象的内容。贡献不一定要很大，一段实验记录、一个更好的问题、一张指标表、一次复现实验的失败记录都可以有价值。

## 可以贡献什么

- 补模块正文
- 补实验设计和实验结果
- 补 benchmark、profiler、压测记录
- 补图表、截图说明和指标解释
- 修正文档里的错别字、坏链接和表述不清的地方
- 整理参考资料，但要说明它解决什么问题

## 写作主线

请尽量沿着这条线组织内容：

```text
成熟系统 -> 观察现象 -> 测量指标 -> 反推瓶颈 -> 剥开机制 -> 生产决策
```

一篇章节或实验记录最好回答这些问题：

1. 用的是哪个成熟系统或现成 workload？
2. 观察到了什么现象？
3. 用哪些指标测？怎么避免测错？
4. 现象背后的瓶颈可能在哪里？
5. 机制是什么？计算、内存、显存、通信、排队、数据管线还是调度？
6. 如果放到一个真实项目里，会怎么取舍？还有什么风险？

## 不要写成什么

不要把内容写成工具清单。

不推荐这样的写法：

```text
这一章介绍 vLLM、TGI、Triton、Ray Serve、KServe 的功能和参数。
```

更推荐这样的写法：

```text
这一章讨论在线推理服务为什么会遇到延迟、吞吐、排队和显存之间的冲突。vLLM、TGI 或 Triton 只是拿来观察这些现象的成熟系统样例。
```

也不要写成八股问答。比如“什么是 KV cache”“什么是 batch size”可以解释，但不要停在定义。最好接着问：它改变了哪些指标？什么时候有用？什么时候会带来新的瓶颈？

## 章节建议结构

可以参考 [章节模板](chapter-template.md)，大致节奏是：

1. 开场问题
2. 最小背景
3. 动手实验
4. 观察结果
5. 指标和瓶颈分析
6. 机制解释
7. 生产取舍
8. 小作业或延伸问题

不需要每次都写得很长。当前阶段更欢迎小而清楚的草稿，后面可以继续补。

## 实验记录要求

实验不追求大规模，优先追求可复现和能解释。

请尽量写清楚：

- 机器环境，比如 CPU、内存、GPU、系统版本
- Python、PyTorch、CUDA、ROCm 或主要工具版本
- 模型、数据、输入长度、图片尺寸、batch size、并发数等关键参数
- warmup、重复次数、统计方法
- 观察到的 latency、throughput、显存、CPU、GPU utilization、P95/P99 等指标
- 哪些结果可信，哪些结果还需要复测

没有 GPU 也可以贡献 CPU 版实验。GPU 版可以作为选做路径，不要把它写成必备条件。

## 参考资料怎么放

参考资料请按问题组织，不按工具堆列表。

可以这样写：

```text
如果想继续理解动态 batch 和队列延迟，可以读这些资料。
```

少写这种：

```text
相关工具：A、B、C、D、E。
```

每条资料最好补一句：它适合回答什么问题，或者它和本章实验有什么关系。

## 状态怎么标

模块状态先用这几类：

- 占位：只有标题或很少说明，还不能当作完整学习材料
- 初稿：已有可读正文，但还需要补实验、图表或参考资料
- 待实验验证：文字方向基本清楚，但实验还没跑通或结果还没整理
- 可学习：正文、实验和必要说明已经比较完整，可以按章节学习

更新模块正文时，如果状态变化，请同步更新 [学习地图](learning-map.md) 和根目录 `README.md` 里的状态表。

## 提交前自查

提交前可以快速看一遍：

- 有没有沿着现象、测量、瓶颈、机制、取舍来写
- 有没有把章节写成工具列表或八股问答
- 有没有暗示课程已经完整，或者承诺真实大规模生产经验
- 实验参数和结果是否能让别人复现或复查
- 新增链接是否能打开
- README 和学习地图里的状态是否一致

# 章节模板

每一章尽量保持同一个节奏，方便共学和写笔记。

```text
1. 开场问题：这一章解决什么困惑？
2. 在 MLSys 地图中的位置：说明本章处在生命周期、系统层次和资源链路的哪一块。
3. 成熟系统：先跑一个真实工具或现成模型的缩减配置。
4. 观察现象：记录指标，不急着下结论。
5. 动手测量：benchmark、profiler、日志或服务指标。
6. 反推瓶颈：计算、内存、IO、通信、排队、调度，哪个更像主因？
7. 剥开机制：只解释和当前现象有关的内部机制。
8. 思维模型：把常见问法整理成可迁移的判断框架。
9. 生产取舍：如果这是线上项目，该怎么选。
10. 小作业：一个必做题，一个选做题。
11. 脚注与参考索引：列出本章用到的书、论文、课程、文档和工程文章。
```

提醒：别急着背，先看现象。

## MLSys 地图位置标准

每个模块都必须写清楚它处在 MLSys 地图里的哪一部分。建议至少回答：

```text
- 生命周期位置：训练 / 推理 / 部署 / 观测 / 数据
- 系统层次：模型代码 / 框架运行时 / 算子与内存 / 单机多卡 / 多机通信 / 服务系统
- 主要资源：计算 / 显存 / 内存 / IO / 网络 / 调度
- 典型症状：慢 / OOM / GPU idle / P95/P99 高 / 多卡不线性 / 通信阻塞
- 关联模块：前置模块、后续模块、横向模块
```

这个小节的目标不是写百科，而是告诉读者：本章回答的是整张系统地图中的哪一类问题。

## 最小背景怎么写

最小背景只补到能看懂现象为止。

如果本章会用到数据结构、计算机组成/体系结构、操作系统、计算机网络、编译原理里的知识，先用简短回顾把关键概念补齐，但不要把它们写成硬性前置门槛。

可以直接写成：

```text
如果学过操作系统和计算机网络，这一节会更顺；没学过也可以先看实验现象，遇到调度、队列、网络延迟时再回来补。
```

## 脚注与参考索引标准

每个模块都必须有“脚注与参考索引”小节。哪怕正文还只是占位，也要先保留这个小节，方便后续补资料。

建议格式：

```md
## 脚注与参考索引

[^ref-1]: 作者或项目名，资料标题，链接。用于说明：这条资料回答本章的哪个问题。
[^ref-2]: 论文或文档标题，链接。用于说明：对应哪个实验、指标或机制。
```

引用资料时尽量说清楚它回答什么问题，不要只堆链接。

# MLSys 课程大纲 v0 讨论稿

这是一份 MLSys 课程草稿。

它不是正式教学大纲，也不是论文目录。更像一份可以边学边补的学习笔记框架，语气轻一点，问题多一点，结论先能用起来。

这门课的主线先放在最前面：

```text
成熟系统 -> 观察现象 -> 测量指标 -> 反推瓶颈 -> 剥开机制 -> 生产决策
```

每一章都尽量围绕这条线走一遍：先用成熟系统跑出一个现象，再测哪里慢，再解释为什么，最后讨论如果放到生产环境里该怎么选。

## 1. 课程定位

一句话版本：面向 AI 工程实践的 MLSys 入门共学课，用任务驱动的方式理解现代 AI 工作负载背后的系统问题。

这里的 MLSys 不是 LLM 专项课。LLM 很重要，但现代 AI 工作负载还包括：

- 视觉模型，比如分类、检测、分割、OCR、视频理解
- 多模态模型，比如图文检索、VLM、文档理解
- 扩散模型，比如文生图、图生图、视频生成
- 推荐、搜索、传统深度学习训练与推理
- 数据处理、模型发布、线上监控、成本评估

课程重点不是把所有底层系统讲成一门硬核体系课，而是帮大家形成生产侧的系统推理能力。

它想做到：

- 用成熟工具和受控实验观察性能现象
- 用指标定位瓶颈
- 用机制解释为什么
- 用案例讨论工程取舍
- 用低算力路径保证大多数同学能跑起来
- 给有 GPU 条件的同学留一条进阶路线

## 2. 这门课不想变成新的八股文

这门课会整理概念，也会总结常见问题，但目标不是制造一套新的 MLSys 背诵材料。

过去很多技术学习容易变成“背框架、背组件、背面试题”。框架会变，工具会变，今天流行的 serving engine、训练框架、MLOps 平台，几年后可能都会换一轮。但系统问题不会消失：

- 计算和内存的矛盾还在
- 吞吐和延迟的矛盾还在
- 显存、通信、调度、排队的问题还在
- 成本、可靠性、可观测性的问题还在
- 从指标判断瓶颈、从瓶颈选择方案的能力还在

所以这门课更想传达一种范式：看到一个新的 ML 系统，不急着背它的 API 和组件名，也不急着从零复刻一个玩具版本，而是先问它的工作负载是什么、资源瓶颈在哪里、指标怎么测、优化在交换什么、生产风险是什么。

如果学生未来遇到新的模型、新的框架、新的硬件、新的 serving 系统，也应该能沿着这套思路继续分析。

## 3. 为什么需要这门课

现在 AI 学习资料很多，但经常是分散的。

模型课会讲 Transformer、Diffusion、CNN、ViT。系统课会讲操作系统、分布式、编译器、数据库。工程教程会讲 Docker、Kubernetes、推理框架、监控。论文笔记会讲训练加速、模型并行、推理优化。

这些资源都很有价值，但新手常常卡在中间：

- 为什么 batch size 变大以后吞吐上去了，延迟也上去了？
- 为什么模型显存够了，训练还是慢？
- 为什么 GPU 利用率看起来很高，请求还是在排队？
- 为什么 notebook 里能跑，做成服务以后就不稳定？
- 为什么量化省了显存，但不一定明显变快？
- 为什么 LLM、diffusion、视觉模型的部署瓶颈不一样？

还有一个很现实的问题：很多 AI Infra、训练平台、推理系统岗位，招聘要求里动不动就写“大规模训练经验”“高并发推理服务经验”“GPU 集群优化经验”。但对大多数学生来说，真实万卡集群、公司级推理平台、线上事故现场都不是随便能接触到的东西。很多高校实验室也没有足够预算让学生长期折腾大规模 GPU 集群，更不用说让学生自己组一套完整的训练、推理和运维环境。

所以这门课不假装让大家拥有真实万卡训练经历，也不鼓励把 demo 包装成生产经验。它更想做的是：用成熟系统的缩减配置和受控实验，把大规模系统里的核心问题缩小到能学习、能复现、能讨论的版本。

比如：

- 用 PyTorch / Hugging Face 上的缩减 workload 观察训练 step、显存、吞吐和 checkpoint
- 用单机或模拟实验理解 batch、通信和并行
- 用成熟 serving 工具或简单服务包装复现延迟、吞吐、排队和 P95/P99
- 用案例讨论成本、SLO、回滚和生产风险
- 用工程报告训练“怎么解释瓶颈和取舍”的表达能力

目标不是让学生背一堆术语去冒充生产经验，而是在没有真实生产权限的情况下，也能建立靠谱的系统直觉和工程表达能力。

已有资源已经覆盖了很多单点知识。这个课程不强调“全新内容”，更想把这些内容按开源共学项目的方式重新打包：

```text
成熟系统 -> 现象观察 -> 动手测量 -> 思维模型 -> 机制解释 -> 生产案例 -> 小作业
```

## 4. 适合谁

适合：

- 学过一点深度学习，想知道模型跑起来以后系统在干什么的同学
- 做过训练或推理 demo，想补性能分析和部署思路的同学
- 对 AI Infra、推理服务、训练平台、MLOps 感兴趣的同学
- 想从“会调包”进阶到“能解释为什么慢，知道怎么改”的同学
- 想理解 AI 系统、机器学习平台、算法工程岗位到底在解决什么问题的同学

默认基础：

- 会写基本 Python
- 知道 PyTorch 或类似框架的基本用法
- 大概知道训练、推理、batch、显存这些词
- 不要求系统课满级
- 不要求有大 GPU 集群

## 5. 这门课不是什么

先把边界说清楚。

它不是：

- LLM 单科课，不会只围绕大语言模型展开
- CUDA 编程硬核课，不会从手写 kernel 开始
- 分布式系统理论课，不会完整覆盖一致性、共识、存储系统
- MLOps 工具大全，不会把所有平台和工具列一遍
- 大模型训练实战营，不承诺真实大规模预训练资源
- 论文精读课，不会按论文时间线逐篇讲

它更像：一份现代 AI 系统学习地图，一套常见性能问题排查笔记，一组低成本可复现的小实验，一套面向生产取舍的思维模型。

## 6. 学习路径

课程分成两条路径。

### 6.1 低算力核心路径

这是默认路径，适合大多数同学。

设备尽量控制在普通笔记本 CPU、免费或低配云环境、单张消费级 GPU。重点不在“跑多大”，而在“看懂现象”。

可以做这些实验：

- 用 PyTorch / Hugging Face 的现成模型看训练吞吐和数据加载瓶颈
- 用 Transformers 或 serving 工具看 batch、序列长度、KV cache 的影响
- 用 diffusers 的缩减配置看采样步数、分辨率和延迟
- 用成熟 serving 工具或服务包装看并发、排队、冷启动、内存占用

### 6.2 进阶 GPU 路径

适合有 GPU 或服务器条件的同学。

可以选做：

- GPU profiling
- 混合精度训练
- 量化推理
- 多卡训练的基本现象
- 推理服务并发与动态 batch
- 简单 Kubernetes 部署
- 监控指标与告警设计

进阶路径不作为主线必做，避免课程变成资源竞赛。

## 7. 贯穿载体系统

这门课尽量不要每章换一套新工具。工具太多，学生会记住名字，忘掉问题。

更推荐选少数几条贯穿线：先用成熟系统跑出真实现象，再一点点剥开内部机制。

原则是：主载体少一点，观察深一点。横向工具按需出现，不把课程变成工具巡礼。

可以先分成三层：

| 层级 | 作用 | 例子 |
| --- | --- | --- |
| 主载体 | 反复使用，承担主要实验和案例 | PyTorch / Hugging Face、vLLM、diffusers |
| 横向工具 | 为了测量、部署、观测、对比而出现 | profiler、benchmark、MLflow、Prometheus / Grafana |
| 扩展参考 | 让学生知道还有别的路线，但不作为主线 | SGLang、TGI、Triton、Ray Serve、KServe、TensorRT-LLM |

### 7.1 PyTorch / Hugging Face 主线

负责训练、普通推理、数据加载、Profiler、Benchmark、dtype、batch size、显存和吞吐。

这条线是默认入口。学生先学会在成熟训练和推理代码上测量，而不是从零写框架。

### 7.2 LLM Serving 主线

建议先选 vLLM 作为主要观察对象，其他 serving engine 放到对比和扩展阅读里。

负责 prefill、decode、KV cache、continuous batching、并发、P95/P99、吞吐和延迟权衡。

这里不是为了背某个 serving engine 的参数，而是通过它看到 LLM 在线服务为什么和普通模型服务不一样。

### 7.3 Diffusion / 多阶段 Pipeline 主线

以 diffusers 这类成熟实现为主要载体。

负责采样步数、scheduler、分辨率、VAE / text encoder / denoiser 等多阶段 pipeline 的耗时拆解。

这条线提醒我们：现代 AI 系统不只有 LLM。扩散模型、视觉模型、多模态模型也有自己的系统瓶颈。

### 7.4 横向工具

MLflow、Prometheus / Grafana、Ray Serve、Triton Inference Server、KServe、Nsight 等工具可以作为横向工具或扩展参考出现。

它们不是课程主角。它们的作用是帮助我们观察、部署、对比和解释现象。

判断一个工具要不要进入主线，可以问：不用它，学生是不是就看不到某个关键现象？如果答案是否，那它就更适合放到扩展参考里。

## 8. 主模块设计

下面是 v0 模块草案。每个模块都尽量从成熟系统里的现象出发，而不是从零手搓一个替代品。

### 模块 0：先搭一张 MLSys 地图
核心问题：AI 系统到底在管什么？
关键词：训练、推理、数据、编译、调度、服务、监控、成本。
要回答：一个模型从 notebook 到线上服务，中间发生了什么？为什么算法同学和平台同学关注点不同？MLSys 和传统后端、传统高性能计算有什么关系？
小实验：跑一个成熟模型或现成 demo，记录 latency、throughput、memory，先建立“一个 AI 系统有哪些可观察表面”。

### 模块 1：工作负载，先看模型在忙什么
核心问题：不同 AI 工作负载为什么系统表现不同？
覆盖对象：视觉模型、Transformer / LLM serving、diffusers pipeline、多模态任务。
要回答：训练和推理的系统压力有什么不同？LLM 的 prefill 和 decode 为什么表现不一样？diffusion 为什么经常被采样步数卡住？视觉模型为什么经常和图片尺寸、预处理有关？
思维模型速记：计算密集、内存带宽密集、batch、序列长度、图片分辨率、参数、激活、KV cache。

### 模块 2：测量，不测就不要猜
核心问题：怎么知道系统慢在哪里？
要回答：latency、throughput、QPS、P50、P95、P99 分别看什么？GPU utilization 高就一定好吗？显存占用、显存峰值、内存碎片有什么区别？为什么 benchmark 结果经常不可信？profiler 里那么多表和 timeline，到底先看哪一列？

这一章要把“会测量”当成核心能力来练，而不是顺手跑一下 `time.time()`。

需要建立的基本习惯：

- 先定义问题，再写 benchmark，不要为了测而测
- 区分 latency、throughput、utilization、memory、cost 这些指标分别回答什么问题
- 知道 warmup、同步、缓存、输入分布、batch size、并发数都会影响结果
- 学会看平均值，也学会看 P95、P99 和长尾
- 学会区分 wall time、CPU time、GPU time、kernel time、data loading time
- 学会用 profiler 找“时间花在哪里”，而不是只凭感觉猜
- 学会写可复现实验：环境、版本、参数、输入、重复次数、统计方法都要记录

建议覆盖的工具和场景：

- Python 基础计时：`time.perf_counter()`、重复运行、均值和方差
- PyTorch 计时：CUDA 同步、`torch.cuda.Event`、`torch.profiler`
- 训练场景：DataLoader 时间、forward、backward、optimizer、GPU idle
- 推理场景：单请求延迟、并发压测、吞吐、排队、P95/P99
- 系统指标：CPU、内存、显存、GPU utilization、显存带宽、磁盘和网络 IO
- 可选进阶：Nsight Systems / Nsight Compute、Triton profiler、服务端 tracing

小实验：围绕一个现成训练或推理 workload 写 benchmark，比较 warmup 前后、同步前后、不同 batch size、不同输入长度、不同并发数的结果。再用 profiler 看一次训练 step 或推理请求，把时间拆成数据准备、模型计算、后处理和等待。

产出：一张“指标怎么读”的速查表，一个可复用的 benchmark 模板，一份 profiler 截图或 trace 解读笔记。

### 模块 3：训练系统，慢在哪里，贵在哪里
核心问题：成熟训练代码跑起来以后，时间和显存到底花在哪里？
要回答：一个训练 step 里有哪些阶段？数据加载为什么会拖慢 GPU？显存主要被哪些东西占掉？混合精度为什么能省显存，有时还能提速？梯度累积和增大 batch 有什么关系？多卡训练为什么会遇到通信瓶颈？
低算力核心：用 PyTorch / Hugging Face 的现成训练脚本或示例，观察 DataLoader worker 数量、batch size、显存峰值、AMP 对吞吐的影响。
进阶路径：DDP 基本实验、checkpointing、ZeRO、pipeline parallel、tensor parallel 的概念理解。

### 模块 4：推理系统，从单次调用到在线服务
核心问题：模型能跑，不等于服务好用。
要回答：单请求延迟和多请求吞吐为什么会冲突？batch、dynamic batching、queueing 分别解决什么？LLM serving 里的 KV cache 为什么重要？diffusion serving 的延迟主要在哪里？视觉模型服务为什么常被图片解码、预处理、后处理影响？
小实验：使用成熟 serving 工具或服务包装运行一个现成模型，用压测工具模拟并发请求，记录 P50、P95、吞吐、内存占用。
思维模型速记：online inference、offline inference、real-time inference、batch inference、冷启动、模型加载、缓存。

### 模块 5：模型压缩与加速，先问为什么再优化
核心问题：量化、蒸馏、剪枝、编译优化到底改了什么？
要回答：FP32、FP16、BF16、INT8 大概差在哪里？量化为什么能省显存，但不一定总是更快？编译优化在优化图、算子、内存，还是调度？ONNX、TensorRT、TVM、Torch Compile 这些名字怎么放到地图里？
低算力核心：在现成模型上比较不同 dtype 的内存和延迟，尝试导出到 ONNX，观察量化前后的模型大小和输出差异。
进阶路径：GPU 上比较 eager、compile、推理引擎的表现，讨论 accuracy、latency、cost 的取舍。

### 模块 6：数据与流水线，别只盯着 GPU
核心问题：AI 系统慢，锅不一定在模型。
要回答：数据读取、解码、增强、tokenization 会不会成为瓶颈？训练前的数据准备和线上特征处理有什么不同？为什么数据版本、特征一致性、样本质量会影响生产系统？多模态任务的数据管线为什么更麻烦？
小实验：使用 Hugging Face Datasets 或现成数据管线，比较图片数据加载和增强耗时、文本 tokenization 批处理、缓存和 streaming 对吞吐的影响。
产出：一份“数据管线排查清单”。

### 模块 7：MLOps 与生产闭环
核心问题：模型上线以后，系统问题才刚开始。
要回答：模型版本、数据版本、代码版本怎么对应？灰度发布和回滚要看哪些指标？线上监控除了 QPS 和错误率，还要看什么？模型效果变差时，怎么区分数据漂移、系统问题和业务变化？成本如何进入工程决策？
小实验：用 MLflow 或类似工具记录实验和模型版本，给一个模型服务添加基础日志和指标，模拟一次新模型上线和回滚。
思维模型速记：model registry、feature store、CI/CD for ML、observability、rollback、cost per request。

### 模块 8：综合案例，按主线走一遍
核心问题：看到一个 AI 应用需求，怎么做系统设计？
候选案例包括图片分类服务、文档问答、文生图 demo、多模态内容审核、离线批处理推理。分析模板固定为工作负载、测量方法、瓶颈判断、机制解释、生产决策。

## 9. 每章模板

每章尽量保持同一个节奏，方便共学和写笔记。

建议模板：

```text
1. 开场问题：这一章解决什么困惑？
2. 最小背景：只补够做实验和理解现象的知识。
3. 动手实验：优先使用成熟工具上的缩减配置，给可选 GPU 版本。
4. 观察结果：记录指标，不急着下结论。
5. 思维模型速记：把常见问法整理成可迁移的判断框架。
6. 机制解释：从系统角度解释为什么。
7. 生产决策：如果这是线上项目，该怎么选。
8. 小作业：一个必做题，一个选做题。
```

每章最好都有一句提醒：别急着背，先看现象。

## 10. Lab 和项目风格

Lab 不追求大而全，优先做小、清楚、能复现。

推荐风格：

- 一个 lab 只验证一个主要问题
- 结果能用表格记录
- 必须有观察，不只贴代码
- 尽量给 CPU 版和 GPU 选做版
- 不依赖真实生产系统
- 不要求大集群
- 优先用成熟工具的缩减配置，不鼓励为了练习而从零造一个替代系统

示例 lab：`batch_size` 对延迟和吞吐的影响，DataLoader worker 数量对训练速度的影响，LLM prefill 和 decode 的时间差异，diffusion 采样步数对生成延迟的影响，图片预处理放在请求内外的差异，serving 工具里的 P95 延迟观察，量化前后模型大小、延迟、输出差异。

每个 lab 的产出可以固定成：实验问题、实验设置、实验结果、现象解释、生产启发。

## 11. 期末项目想法

期末项目不要求做成完整平台，重点是把主线跑通。

可选方向：

1. 成熟模型推理服务性能分析：基于现成模型和 serving 工具，压测不同并发和 batch，给出部署建议。
2. LLM serving 实验：比较 prefill、decode、KV cache 相关现象，讨论延迟、吞吐和显存取舍。
3. Diffusion 生成服务排队实验：用 diffusers 或现成 pipeline 的缩减配置，比较单请求、队列、限流策略。
4. 训练性能排查报告：基于现成训练脚本分析数据加载、batch size、AMP、显存，输出优化建议。
5. 多模态流水线案例：比如 OCR 加文本分类，图片 embedding 加检索，分析每个阶段耗时。
6. MLOps 闭环实验：记录模型版本，做简单发布和回滚，设计基础监控指标。

项目报告建议结构：背景和目标、工作负载描述、测量方法、关键结果、瓶颈分析、机制解释、生产决策建议、还没解决的问题。

引用资料时建议按“问题”组织，而不是按“工具”组织。

比如不要写成：这一章学习 Triton、vLLM、Ray Serve、KServe。

更推荐写成：这一章讨论在线推理服务怎么处理并发、排队、batch 和资源隔离。不同工具只是解决方案样例。

## 14. v0 待讨论问题

这份大纲还需要继续讨论：每章选哪些成熟模型、数据集和工具配置，是否统一使用 PyTorch，GPU 选做部分到什么深度，LLM、diffusion、视觉、多模态的比例怎么分配，是否单独安排 Kubernetes 和容器，期末项目是个人还是小组，哪些内容先做成基础篇，哪些内容留给进阶篇持续补充。

当前 v0 的原则是：不做工具动物园，不做 LLM 单科课，不装成大规模生产系统实战，不把门槛抬太高。先让大家能测、能解释、能做基本生产判断。

如果每章都能把“成熟系统 -> 观察现象 -> 测量指标 -> 反推瓶颈 -> 剥开机制 -> 生产决策”走一遍，这门课就有了自己的骨架。
